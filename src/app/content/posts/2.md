Ever wondered how Medium calculates the "read time" for articles? In this three-part series, you'll learn how to implement a privacy-friendly solution that tracks user engagement without cookies or consent banners. Using Vue.js for tracking, Plausible.io for analytics, and Azure Functions for reporting, we'll guide you step-by-step to create a seamless read-time analytics system for your site.

<!--more-->

**Part 1**: We explore how Medium.com calculates "read time" and implement a tracking solution to measure reading progress.

**Part 2**: We set up Plausible.io for analytics without requiring cookie consent and configure goals for visual insights.

**Part 3**: We use Azure Functions to interact with the Plausible.io API and retrieve data on how many users have fully read the article.

**Example**: To provide a clear demonstration, your current reading progress will be displayed prominently at the top of the page.

This is a series of 3 posts that walks you through on how to create cookie-less read time analytics, respecting your users' privacy and not bothering them with an annoying cookie consent banner, all the while still providing valuable analytics about your site.
In part 1: we discuss how medium.com calculates the "read time" and how we can implement tracking the read time of your page.
In part 2: we'll setup Plausible.io to have analytics without cookie consent banner and setup goals to have visual insights
And in part 3: we'll use azure functions to call use the Plausible.io api and get the actual amount of people that read our article.

Now that we have the exact percentage of read and scroll progress, let's create another composable that transforms this data in a former that is easier to report.
Let's use 6 different statuses to indicate how far the user got, reading the page:
opened: default status

peeked: 10% of the time or scroll

quarter-read: 25%

half-read: 50%

three-quarter-read: 75%

read: 100%

When tracking an event we use one of these statuses, each time the progress changes, we log an event with the same name of the status. Resulting in a nice funnel when counting only the unique events per visitor.
← Image of Funnel →
To change the current status, both the time as the scroll progress need to have reached the thresholds. So, we only get to the status 'peeked' when we scrolled 10% of the page, but also have spent 10% of the time needed to read the article. By the time we logged the 'read' status, we know the user has spent 100% of the 'read time' and scrolled 100% of the page.
